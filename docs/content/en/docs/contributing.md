Thank you for your interest in contributing to `gofhir/models`. This page covers the development setup, testing workflow, and guidelines for submitting changes.

## Prerequisites

- **Go 1.23 or later** -- The project uses Go workspaces and generics.
- **Git** -- For cloning the repository and managing branches.

## Getting Started

### Clone the Repository

```bash
git clone https://github.com/gofhir/models.git
cd models
```

### Understand the Project Layout

```
models/
  cmd/generator/     # Code generator tool
  internal/          # Shared generation infrastructure
  specs/             # FHIR StructureDefinition JSON files
  r4/                # Generated R4 module (github.com/gofhir/models/r4)
    helpers/          # Hand-written helper constants (not generated)
  r4b/               # Generated R4B module (github.com/gofhir/models/r4b)
  r5/                # Generated R5 module (github.com/gofhir/models/r5)
  docs/              # Documentation site (Hugo)
  go.work            # Go workspace file
```

### Set Up the Workspace

The repository uses a Go workspace (`go.work`) to link all modules together for local development. No additional setup is required -- Go will automatically use the workspace when you build or test from within the repository.

```bash
# Verify the workspace is functional
go work sync
```

## Running Tests

Each FHIR version module has its own test suite. Run tests from the module directory:

```bash
# Test R4
cd r4 && go test ./...

# Test R4B
cd r4b && go test ./...

# Test R5
cd r5 && go test ./...
```

Or run all tests from the repository root:

```bash
cd r4 && go test ./... && cd ../r4b && go test ./... && cd ../r5 && go test ./...
```

The test suites cover:

- Resource struct construction and field access
- JSON marshaling and unmarshaling (including `resourceType` injection and contained resources)
- XML marshaling and unmarshaling (namespace handling, primitive encoding, backbone elements)
- Fluent builder correctness
- Functional options correctness
- Registry functions (NewResource, UnmarshalResource, GetResourceType, etc.)
- FHIRPath model metadata accuracy
- Code system constant validation
- Decimal type precision preservation

## Regenerating Models

If you modify the code generator or update the FHIR specification files, regenerate the code:

```bash
# Regenerate R4 code
go run cmd/generator/main.go r4

# Regenerate R4B code
go run cmd/generator/main.go r4b

# Regenerate R5 code
go run cmd/generator/main.go r5
```

After regeneration, run the full test suite to verify correctness:

```bash
cd r4 && go test ./... && cd ../r4b && go test ./... && cd ../r5 && go test ./...
```

{{< callout type="info" >}}
Never edit the generated files in `r4/`, `r4b/`, or `r5/` directly (except for the `helpers/` subdirectory). All generated files begin with `// Code generated by gofhir. DO NOT EDIT.` and will be overwritten when the generator runs. Modify the generator code in `internal/codegen/generator` instead.
{{< /callout >}}

## Types of Contributions

### Bug Fixes

1. Open an issue describing the bug with a minimal reproduction case.
2. Create a branch: `git checkout -b fix/description`.
3. Add a failing test that demonstrates the bug.
4. Fix the issue -- if it involves generated code, fix the generator and regenerate.
5. Verify all tests pass.
6. Submit a pull request.

### New Features

1. Open an issue to discuss the feature before starting work.
2. Create a branch: `git checkout -b feat/description`.
3. Implement the feature with tests.
4. If the feature involves generated code, update the generator templates and regenerate all versions.
5. Submit a pull request.

### Documentation

1. Documentation lives in `docs/content/en/docs/`.
2. The site uses Hugo with the Hextra theme.
3. To preview locally:

```bash
cd docs
hugo server
```

### Helper Constants

The `r4/helpers/` directory contains hand-written convenience constants. Contributions of new helper constants are welcome -- for example:

- Additional LOINC codes for common clinical observations
- Additional UCUM unit helper functions
- Pre-built `CodeableConcept` values for common value sets

## Commit Message Convention

This project uses [Conventional Commits](https://www.conventionalcommits.org/) for commit messages. The release-please bot uses these to generate changelogs and determine version bumps.

| Prefix | Purpose | Version Bump |
|--------|---------|:------------:|
| `feat:` | New feature | Minor |
| `fix:` | Bug fix | Patch |
| `refactor:` | Code restructuring | -- |
| `docs:` | Documentation only | -- |
| `test:` | Test additions/changes | -- |
| `chore:` | Maintenance tasks | -- |

Examples:

```
feat: add Encounter resource builder validation
fix: correct XML namespace for contained resources
refactor: consolidate generated files and add XML deserialization
docs: add examples for Bundle construction
```

## Pull Request Guidelines

1. **One concern per PR** -- Keep pull requests focused on a single change.
2. **Include tests** -- All code changes should have corresponding tests.
3. **Run the full suite** -- Ensure all tests pass across all three FHIR versions before submitting.
4. **Regenerate if needed** -- If your change affects the generator, include the regenerated output in the PR.
5. **Follow existing style** -- Match the coding style and patterns used in the existing codebase.
6. **Update documentation** -- If your change adds or modifies public API, update the relevant documentation pages.

## Code of Conduct

Please be respectful and constructive in all interactions. We are building tools for healthcare interoperability, and a welcoming community helps us do that better.
